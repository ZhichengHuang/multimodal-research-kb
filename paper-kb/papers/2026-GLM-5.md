## 一句话总结
744B 参数（40B 激活）256 专家 MoE 模型，通过 DeepSeek Sparse Attention（DSA）实现长上下文高效推理，采用多阶段 RL 管线（Reasoning RL + Agentic RL + General RL + 跨阶段蒸馏）和全异步解耦 RL 框架（Slime），在编码和 agentic 任务上达到开源 SOTA。

## 核心 Insight
1. **多阶段 RL 管线 + 跨阶段蒸馏防止遗忘**: 将 RL 训练拆分为 Reasoning → Agentic → General 三个专用阶段，每阶段聚焦特定能力，最后通过 on-policy cross-stage distillation 将各阶段最优策略合并到统一模型，解决顺序 RL 训练的灾难性遗忘问题。
2. **IcePop 解决 RL 训练-推理不匹配**: 在 GRPO 中区分训练策略 π_train 和推理策略 π_infer，通过显式 mismatch suppression 消除两者差异导致的性能退化——对 dLLM 的 masked diffusion 训练-推理不匹配问题有直接启示。
3. **全异步解耦 Agentic RL**: Slime 框架将推理引擎和训练引擎完全解耦，支持 1000+ 并发 rollout，Token-in-Token-out (TITO) 网关保证 tokenization 一致性，解决了长时间 agent rollout 的效率瓶颈。

## 与已有工作的关系
- **继承自**: DeepSeek-V3（DSA 注意力机制）、GRPO（RL 算法基础）
- **对比**:
  - vs [[2025-KimiK2.5]]: 同为大规模 MoE RL 管线，但 GLM-5 更侧重 agentic RL 基础设施（Slime 框架）而非多模态视觉能力
  - vs [[2025-OpenMMReasoner]]: GLM-5 的 IcePop 技术与 GSPO 都解决 RL 训练稳定性，但从不同角度（训练-推理匹配 vs 组别内排序）
- **互补**:
  - 与 dLLM RL 工作（[[2025-MMaDA]], [[2026-LaViDa-R1]], [[2026-LFPO]]）: GLM-5 的 IcePop、异步 RL 基础设施、跨阶段蒸馏可迁移到 dLLM RL 管线

## 方法细节

### 架构
- **规模**: 744B 总参数, 40B 激活参数, 256 专家, 80 层
- **前代**: GLM-4.5 为 355B 总 / 32B 激活
- **Multi-Latent Attention (MLA)**: 采用 Muon Split（分拆矩阵为不同 head 的独立子矩阵做正交化），使 MLA 匹配 GQA-8 性能；head 维度从 192 增至 256 同时减少 1/3 head 数量，训练计算不变但降低解码计算
- **DeepSeek Sparse Attention (DSA)**: 动态细粒度 token 重要性选择，替代 O(L²) 密集注意力，长序列降低 1.5-2× 注意力计算，无信息损失。DSA 训练: 1000 步 warm-up + 20B tokens 稀疏适配。RL 中使用 deterministic top-k（torch.topk 替代非确定性 CUDA 实现）确保一致输出
- **Multi-Token Prediction (MTP)**: 3 层 MTP 共享参数训练（保持单层内存一致性），acceptance length 2.76 vs DeepSeek-V3.2 的 2.55

### 预训练
- **总 token**: 28.5T
- **上下文扩展**: 4K → 32K (1T tokens) → 128K (500B) → 200K (50B)
- **代码数据**: 10M issue-PR pairs, ~160B unique tokens, 跨 9 种编程语言
- **数据质量**: DCLM 句子嵌入分类器 + World Knowledge 分类器（长尾信息）
- **量化**: SFT 阶段 INT4 QAT（量化感知训练），训练/推理共用量化核

### 多阶段 RL 管线

**Stage 1: SFT**
- 通用对话 + 推理 + 编码/Agent 三类数据
- 最长上下文 202,752 tokens
- Thinking 模式: Interleaved Thinking（每次回复/工具调用前思考）+ Preserved Thinking（跨轮次保留思维块）

**Stage 2: Reasoning RL**
- 算法: GRPO + IcePop
- IcePop 解决训练-推理不匹配: L(θ) = -E[1/G ∑(1/|y_i|) ∑ pop(ρ, 1/β, β) · min(r·Â, clip(...)·Â)]
- 超参: β=2, ε_low=0.2, ε_high=0.28, group_size=32, batch_size=32
- 领域: 数学、科学、代码（竞赛编程+科学计算）、工具集成推理 (TIR)
- DSA RL 适配: deterministic top-k + 冻结 indexer 参数

**Stage 3: Agentic RL（Slime 框架）**
- 全异步解耦: 推理引擎和训练引擎通过 HTTP API 分离
- Multi-Task Rollout Orchestrator: 1000+ 并发 rollout
- Token-in-Token-out (TITO) 网关: 保证精确 tokenization，消除重新编码不匹配
- 直接双边重要性采样: token 级 clipping [1-ε_ℓ, 1+ε_h]
- 严重 off-policy 样本过滤: age 阈值 τ
- DP-aware routing: KV cache locality 优化
- 环境建设:
  - SWE: 10,000+ 可验证环境，9 种语言（RepoLaunch 框架）
  - Terminal: 三阶段合成（任务草案→Harbor 实现→迭代优化），Docker 构建准确率 >90%
  - Search: Web Knowledge Graph（200万+ 高信息网页），多跳 QA 对

**Stage 4: General RL**
- 多维目标: 基础正确性 + 情感智能 + 任务特定质量
- 混合 reward: 规则 reward + ORM (Outcome Reward Model) + GRM (Generative Reward Model)
- Human-in-the-loop: 专家人类响应作为风格锚点，防止模型收敛为公式化模式

**Stage 5: On-Policy Cross-Stage Distillation**
- 用前几阶段的教师模型做 on-policy 蒸馏
- Advantage 替换为: Â_i,t = sg[log(π_teacher_infer / π_train)]
- group_size=1, batch_size=1024（提高数据吞吐）

### Slime 框架基础设施
- **灵活 Rollout 接口**: 支持多轮交互、工具调用、环境反馈、verifier 分支
- **延迟优化**: FP8 rollout 推理 + MTP 减少尾延迟 + Prefill-Decode 分离防止多轮 RL 中预填充干扰解码
- **容错**: 心跳驱动故障容忍——定期检测不健康服务器，主动终止和注销

## Building Blocks（可复用组件）

### Block 1: IcePop（训练-推理不匹配矫正）
- **做法**: 在 GRPO 目标函数中显式区分训练策略 π_train 和推理策略 π_infer，通过 pop 函数抑制两者的分布偏差。β=2 控制抑制强度，ε_low=0.2/ε_high=0.28 非对称 clipping
- **机制 (WHY it works)**: RL 训练时策略持续更新（on-policy 理想），但实际 rollout 使用旧策略（off-policy 现实）。IcePop 通过重要性权重校正显式补偿这一偏差，而非简单丢弃 off-policy 样本（浪费计算）或忽略偏差（引入 bias）
- **适用条件**: 异步 RL 训练中 rollout 策略与训练策略存在版本差异; 训练时间远长于 rollout 时间的场景
- **什么时候会 break**: π_train 和 π_infer 差异过大（age 极高）时重要性权重方差爆炸; β 和 ε 超参需要针对不同任务调优
- **可组合方向**: 可直接应用于 dLLM RL（[[2025-MMaDA]] UniGRPO、[[2026-LaViDa-R1]] answer-forcing GRPO）中的异步训练场景; 与 [[2026-EBPO]] 的 shrinkage baseline 正交组合

### Block 2: 全异步解耦 Agentic RL 框架（Slime）
- **做法**: 推理引擎和训练引擎通过标准 HTTP API 完全解耦，Multi-Task Rollout Orchestrator 管理 1000+ 并发 rollout。TITO 网关保证 token 流精确一致。心跳驱动容错自动处理节点故障
- **机制 (WHY it works)**: Agent rollout 时间极长（分钟级 vs 秒级文本生成），同步训练会导致 GPU 大量空闲。完全解耦允许训练引擎持续消费已完成的 rollout 数据，而 rollout 引擎持续生成新轨迹，最大化 GPU 利用率。TITO 避免了重新 tokenize 时因 BPE 合并歧义导致的 token 序列不一致
- **适用条件**: 长时间 rollout 的 RL 训练（agentic 任务、多轮交互、tool-use）; 大规模集群（需要容错）
- **什么时候会 break**: rollout 极短（纯文本 QA）时解耦开销可能大于收益; 单节点小规模训练不需要这种复杂基础设施
- **可组合方向**: 可为 dLLM 统一模型的多任务 RL（[[2025-Lumina-DiMOO]] Self-GRPO、[[2026-LaViDa-R1]] 多任务 RL）提供基础设施模板; 特别适合 dLLM 图像生成 RL（rollout 包含多步去噪，时间远长于 AR 文本生成）

### Block 3: 多阶段 RL + On-Policy Cross-Stage Distillation
- **做法**: 将 RL 拆分为 Reasoning → Agentic → General 三个专用阶段，每阶段聚焦特定能力域。最终通过 on-policy cross-stage distillation 合并——用各阶段最优教师模型的推理策略作为 advantage 信号（Â = sg[log(π_teacher / π_train)]），group_size=1 消除多样本采样需求
- **机制 (WHY it works)**: 顺序 RL 训练面临灾难性遗忘（后阶段覆盖前阶段能力）。跨阶段蒸馏通过保留各阶段教师的策略知识，在最终模型中恢复早期阶段的能力。group_size=1 + batch_size=1024 的设计将蒸馏转化为大批量监督学习（用教师 log prob 差作为优势），计算效率高
- **适用条件**: 需要训练多种异质能力（推理/编码/agent/对齐）; 各能力域有不同的最优 RL 策略
- **什么时候会 break**: 各阶段能力之间存在根本性冲突（非遗忘而是干扰）时蒸馏无法调和; 教师模型质量参差导致蒸馏目标噪声大
- **可组合方向**: 可应用于 dLLM 统一模型的多任务 RL——理解 RL + 生成 RL + grounding RL 分阶段训练后蒸馏合并; 与 [[2025-Lumina-DiMOO]] Self-GRPO 的联合目标形成对比（联合优化 vs 分阶段蒸馏两种范式）

### Block 4: 混合 Reward 系统（Rule + ORM + GRM）
- **做法**: 在 General RL 阶段使用三类 reward 的组合——规则 reward（格式正确性、基本约束）、Outcome Reward Model（ORM，结果正确性评估）、Generative Reward Model（GRM，生成式多维评估包括正确性/情感智能/任务质量）。同时引入专家人类响应作为风格锚点
- **机制 (WHY it works)**: 规则 reward 提供无噪声基础信号（格式/约束违反直接惩罚）; ORM 提供结果级评估（较稳定但粗粒度）; GRM 提供过程级多维评估（细粒度但可能有噪声）。三者互补覆盖不同评估维度。风格锚点防止 reward hacking 导致的公式化输出
- **适用条件**: 需要平衡多维目标（正确性+风格+安全性）; 有足够资源训练/部署 ORM+GRM
- **什么时候会 break**: 三类 reward 信号冲突时（如 GRM 鼓励长回答但规则要求简洁）可能导致训练不稳定; GRM 的训练质量直接影响整体效果
- **可组合方向**: GRM 概念与 [[2025-KimiK2.5]] 的 GRM 一致，两者互相验证; 可替代 dLLM RL 中的简单 CLIP+ImageReward reward（[[2025-MMaDA]]）提供更细粒度信号; 与 [[2025-Lumina-DiMOO]] Self-GRPO 的自评估互补（外部评估器 vs 自评估）

### Block 5: DSA 在 RL 中的适配（Deterministic Top-k + Frozen Indexer）
- **做法**: 在 RL 训练中采用 deterministic top-k operator（torch.topk 替代非确定性 CUDA 实现）确保 DSA 的 token 选择一致性; 默认冻结 DSA 的 indexer 参数
- **机制 (WHY it works)**: RL 训练中策略优化依赖精确的梯度信号。非确定性 CUDA top-k 在相同输入下可能选择不同 token 集合，导致前向传播不一致→advantage 估计噪声增大→训练不稳定。冻结 indexer 避免 sparse attention pattern 在 RL 训练中漂移
- **适用条件**: 使用 sparse attention 机制的模型做 RL 训练; 需要训练-推理结果一致性
- **什么时候会 break**: sparse attention 的 token 选择模式需要随 RL 训练调整时冻结 indexer 可能过于保守
- **可组合方向**: 对 dLLM 的推理加速方案（[[2025-Sparse-LaViDa]] 的 token 截断、[[2025-Lumina-DiMOO]] ML-Cache）在 RL 训练中的适配有参考价值——RL 训练中的确定性和一致性需求可能需要类似的适配策略

## Anti-patterns / 已知失败模式
- **非确定性 CUDA 算子破坏 RL 训练**: DSA 中非确定性 top-k 导致 RL 训练不稳定，必须用 deterministic 替代。对所有在 RL 训练中使用 sparse/dynamic 计算的方案的警示
- **同步 Agentic RL 的 GPU 浪费**: Agent rollout 时间远长于文本生成（分钟级 vs 秒级），同步训练导致 GPU 大量空闲等待。必须解耦推理和训练
- **顺序 RL 阶段的灾难性遗忘**: 依次训练 Reasoning→Agentic→General 会导致后阶段覆盖前阶段能力，需要跨阶段蒸馏恢复
- **BPE 重新编码不一致**: Agent rollout 中工具返回的文本经重新 tokenize 后可能与原始 token 序列不一致（BPE 合并歧义），TITO 网关通过保持原始 token 流解决
- **Prefill-Decode 干扰**: 多轮 RL 中长前缀的预填充会抢占解码资源，需要 Prefill-Decode 分离

## 问题定位
### 解决的问题
- **异步 Agentic RL 的系统级瓶颈**: Slime 框架完整解决了长 rollout（分钟级 agent 交互）下 GPU 利用率低的问题，提供 1000+ 并发 rollout + 心跳容错 + TITO tokenization 一致性的全套方案
- **RL 训练-推理策略不匹配**: IcePop 通过显式重要性权重校正解决异步 RL 中旧 rollout 策略与当前训练策略的分布偏差
- **顺序 RL 的灾难性遗忘**: On-policy cross-stage distillation 将各阶段最优教师合并到统一模型，保留多种能力
- **Sparse attention 在 RL 中的不稳定性**: 发现并解决非确定性 CUDA top-k 破坏 RL 训练稳定性的问题

### 未解决的问题
- 问题: IcePop 的超参（β=2, ε_low=0.2, ε_high=0.28）对不同任务/架构的敏感性
  - 为什么难: 最优超参取决于策略漂移速率，不同任务/架构下漂移行为不同
  - 潜在思路: 自适应 β（基于 rollout age 的实时 KL divergence 估计动态调整）

- 问题: 跨阶段蒸馏中各教师模型质量参差导致的蒸馏目标噪声
  - 为什么难: 不同阶段的教师在不同能力维度上最优，简单合并可能互相稀释
  - 潜在思路: 加权蒸馏（根据教师在特定领域的 reward 置信度动态调整权重）

- 问题: Slime 框架在中等规模集群（32-64 GPU）下的效率
  - 为什么难: 框架为 744B 模型设计，overhead-to-computation 比例在小模型上可能不利
  - 潜在思路: 轻量化 Slime（减少 HTTP 通信开销，共享进程替代独立服务）

### 对问题树的推进
- 推进了 [[problem-tree#RL-2a]] 🔴→🟡（系统层面）: Slime 异步框架从系统吞吐角度降低 RL 采样成本，与 KB 已有方法（UniGRPO 结构化 mask、complementary masking、LFPO 无似然度）正交——已有方法降低单样本成本，Slime 提升多样本并发效率
- 推进了 [[problem-tree#RL-3a]] 🟡（补充 reward 架构）: 混合 Reward（Rule+ORM+GRM）+ 人类风格锚点，提供比 CLIP+ImageReward 更成熟的多维 reward 方案
- 推进了 [[problem-tree#RL-3c]] 🟡→🟡（AR→dLLM 技术迁移清单）: IcePop、跨阶段蒸馏、DSA RL 适配三项技术可直接迁移到 dLLM RL，丰富了 AR↔dLLM 方法论交叉点
- 推进了 [[problem-tree#Uni-5a]] 🟡（新范式选项）: 跨阶段蒸馏为 dLLM 统一模型的多能力 RL 提供第三种范式（vs DiMOO 联合优化、LaViDa-R1 统一 PG）
- 新增问题 [RL-2i]: IcePop 解决策略版本不匹配，但 dLLM 还有输入分布不匹配（P-Diff-04）——是否可将 IcePop 扩展为双维度修正？
- 新增问题 [RL-2j]: 跨阶段蒸馏在异质模态任务（text NLL<2 vs image NLL>6）间的 advantage 归一化策略
- 新增问题 [RL-2k]: RL 训练中 sparse/dynamic 计算的确定性需求 vs dLLM 推理加速的动态性冲突

## 个人深度评注

> 以下由 Critic + Ideator Agent 联合分析

### 一、对 KB 的核心价值判定

GLM-5 不是多模态论文，而是一篇**纯文本 LLM + Agentic RL 基础设施论文**。对本 KB（聚焦多模态 dLLM）的价值不在于其最终性能，而在于**三项可迁移的 RL 方法论组件**:

| 组件 | 迁移目标 | 价值判定 |
|---|---|---|
| IcePop | dLLM 异步 RL 的策略偏差校正 | **高** — 架构无关，直接嵌入任何 GRPO 变体 |
| Cross-Stage Distillation | dLLM 多能力 RL 的遗忘防止 | **中高** — 需要模态感知归一化适配 |
| Slime 异步框架 | dLLM 长 rollout RL 的 GPU 利用率 | **中** — 工程价值大但适配成本高 |
| DSA RL 确定性适配 | dLLM 推理加速在 RL 中的稳定性 | **中** — 警示性发现，对 Sparse-LaViDa/ML-Cache 有直接影响 |
| Hybrid Reward | dLLM T2I RL reward 设计 | **低中** — 概念与 Kimi K2.5 GRM 重复 |

### 二、IcePop 对 dLLM 的特殊启示

dLLM 存在 **双重** 训练-推理不匹配:
1. **策略版本不匹配**（与 AR 相同）: 异步 RL 中 rollout 用旧策略、训练用新策略 → IcePop 直接解决
2. **输入分布不匹配**（dLLM 独有）: 训练时 context 为干净数据，推理时 context 包含模型自身预测的（可能错误的）token → IcePop 无法解决，需要 ReDiff 精炼训练或 XDLM 混合噪声核

**潜在组合**: IcePop（修正第一重不匹配）+ XDLM k=0.1（缓解第二重不匹配）+ LFPO（避免似然度估计方差），形成 dLLM RL 的"三重稳定化"方案。这是 KB 中首次识别出的可组合三层修正策略。

### 三、跨阶段蒸馏 vs 联合优化的 tradeoff

| 维度 | 联合优化 (DiMOO Self-GRPO) | 分阶段 + 蒸馏 (GLM-5 模式) |
|---|---|---|
| 梯度冲突 | 存在（image/text 梯度竞争） | 无（各阶段独立训练） |
| 跨模态协同 | 天然实现 (P-Uni-01) | 需要蒸馏阶段恢复 |
| 训练成本 | 1x | 2.5-3x |
| 超参复杂度 | 单套超参 | 每阶段+蒸馏阶段各一套 |
| NLL 尺度问题 | 严重（KL 在 image NLL>6 时不稳定） | 可分别处理 |

**判断**: 对当前 8-10B dLLM，联合优化的简洁性更有价值; 但随着 dLLM 规模扩展到百亿+参数、任务更异质，跨阶段蒸馏可能成为必要策略。GLM-5 在 744B 规模验证了这一点。

### 四、DSA RL 适配的警示意义

GLM-5 发现非确定性 CUDA top-k 破坏 RL 训练稳定性。KB 中五种 dLLM 推理加速方案（Sparse-LaViDa token 截断、ML-Cache logit 缓存、MARS-Cache 帧级锚点、Prefix-DLM KV 缓存、Block Diffusion 块内并行）均使用动态/非确定性计算。**如果这些方案在 RL rollout 中激活，可能引入相同的不稳定性**。建议:
- RL rollout 阶段关闭所有动态加速方案（"clean rollout"策略）
- 或为每种加速方案设计 deterministic 变体（如 Sparse-LaViDa 的截断比例固定化、ML-Cache 的 cache_ratio 固定化）
- 这构成一个新的研究方向: **RL-safe inference acceleration for dLLM**
